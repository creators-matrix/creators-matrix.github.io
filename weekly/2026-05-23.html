一天写了8个工具，踩了3个坑

2026-05-23

今天可能是这个网站上线以来最高产的一天。从上午十点干到凌晨一点，一共上线了8个新工具。但过程远没有数字看起来那么顺利——踩了3个让我想砸键盘的坑。

做了什么

第一批：4个开发者刚需工具

  • JSON格式化校验器：美化、压缩、语法校验三种模式。这个是我最常用的——调试API时对着压缩成一行的JSON找bug，「少了一个逗号查了一个小时」的痛。
  • Base64编解码工具：文本和图片都能编解码。不用再找在线工具上传文件了，全部在浏览器本地完成。
  • Unix时间戳转换器：秒级/毫秒级自动识别，实时显示当前时间戳。起因是「对接第三方API时对方用秒级、我们用毫秒级，一个少三个零的bug查了半天」。
  • 正则表达式测试器：输入正则和测试文本，实时高亮匹配结果，显示捕获组。写了一键加载的日期/邮箱/URL示例。

第二批：3个差异化工具

  • AI Prompt效果对比器：左右分栏对比两个Prompt，自动分析Token消耗、关键词密度、结构完整性评分。没办法真接API，但可以帮你快速判断哪个Prompt更「专业」。
  • Markdown实时预览编辑器：左边写、右边看，带工具栏。起因是「写README时在VSCode里写完还得push到GitHub才能看到渲染效果」。纯手写的Markdown解析器，没有引入任何第三方库。
  • 中医方剂加减计算器：选了桂枝汤、麻黄汤、小青龙汤、半夏泻心汤四个经典方剂，可以加减药味、调整剂量，自动计算总剂量和药量占比。用柱状条可视化每味药的占比。

第三批：代码片段收藏夹 + 全站使用计数器

  • 代码片段收藏夹：用localStorage存储，支持搜索过滤、编辑删除、一键复制。所有数据存在浏览器里，不需要登录。
  • 全站使用计数器：给14个工具页面都注入了localStorage计数器。每个工具页底部显示「🔢 使用 X 次」。实现方式是Python脚本批量注入，因为涉及14个文件手动改会疯。

踩了什么坑

  1. macOS的 sed 和 Linux 不一样，批量替换翻车
    给14个工具页注入计数器时，写了一个bash脚本用 sed 批量替换。在Linux上测试通过的 sed 命令，到了macOS上完全不行——BSD sed 对特殊字符的处理和 GNU sed 不一样，`|` 分隔符跟替换内容里的管道符冲突，所有文件替换失败但没有报错退出。最后改用 Python 的 `os.walk()` + `str.replace()` 才搞定。教训：在macOS上写脚本，能不用sed就别用sed，Python更可靠。
  2. macOS的 seq 1 0 居然有输出
    给全站35个HTML文件加favicon时，用了一个脚本根据文件深度计算相对路径。里面用了 `seq 1 $depth` 来生成 `../` 前缀。当depth=0时，在Linux上 seq 1 0 不输出任何东西（正确行为），但在macOS上 seq 1 0 输出 `1` 和 `0` 两行，导致depth=0的文件路径变成了 `../../assets/avatar.png` 而不是 `./assets/avatar.png`。所有根目录文件的favicon路径全错。这个bug花了20分钟才定位到是 seq 的行为差异——谁能想到 seq 1 0 会有输出？
  3. 五运六气算法天干→五行映射全错（这是昨天发现的，但值得记下来）
    一位中医背景的读者私信指出：推算器里天干到岁运的映射用的是「天干自身五行」（甲乙木、丙丁火...），但《黄帝内经》五运六气用的是「天干化运」（甲己化土、乙庚化金、丙辛化水、丁壬化木、戊癸化火）。10个映射错了8个，只有己和庚碰巧对上了。如果不是有人指出，这个bug可能永远没人发现——因为雷达到看起来「有模有样」，非专业人士根本看不出来。

感受

今天写了大概3000行HTML+JS，但最有价值的不是代码量，是踩坑的过程。两个操作系统兼容性bug，一个算法原理性错误——每一个都让我意识到「以为自己懂」和「真的懂」之间的差距。

不过话说回来，一天上线8个工具，放在失业前的公司里，光是需求评审和排期就要两周。
